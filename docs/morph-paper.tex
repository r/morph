\begin{abstract}
Modern AI-centric software development---involving large language model calls, retrieval-augmented generation, tool use, and agent-driven workflows---breaks the assumptions underlying traditional version control. Git tracks deterministic source code where identity is byte equality, reproducibility means identical output, and merge is syntactic reconciliation. When the versioned artifact is a \emph{transformation program} whose operators are stochastic, effectful, and environment-dependent, those assumptions stop working. This paper introduces \textsc{Morph}, a distributed version control system designed for transformation programs. \textsc{Morph} extends Git's content-addressed Merkle DAG architecture with three primitives: (1)~\emph{programs}---directed acyclic graphs of typed operators including prompt calls, tool invocations, and deterministic transforms; (2)~\emph{evaluation suites}---versioned behavioral contracts specifying metrics, thresholds, and certification rules; and (3)~\emph{runs}---immutable execution receipts recording full environment, trace, and metric observations. A \textsc{Morph} commit bundles a program with an evaluation contract and a certified metric vector. Merge requires the candidate to dominate the componentwise join of both parents' certified scores---making behavioral regression a merge conflict. We formalize programs as effectful computations in a monadic framework, define behavioral equivalence via a preorder on certificate vectors, and prove that merge-as-dominance preserves monotonic behavioral improvement along any branch. We describe a concrete v0 system and discuss implications for agent-driven software development, reproducibility, and human-AI collaboration.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Software development is changing faster than the tools we use to manage it. Large language models are not autocomplete for code anymore; they are components inside larger systems that retrieve context, call tools, generate and modify code, run tests, and iterate~\citep{yao2023react, jimenez2024swebench}. These \emph{transformation programs}---pipelines combining prompt calls, retrieval, tool invocations, and deterministic transforms---differ from traditional source code in ways that break the foundations of how we track changes.

Git~\citep{torvalds2005git}, the dominant version control system, rests on three assumptions: (i)~identity is byte equality, (ii)~reproducibility means identical output bytes, and (iii)~merge is syntactic text reconciliation. All three break when the versioned artifact is a stochastic, environment-dependent transformation pipeline:

\begin{itemize}[nosep]
  \item \textbf{Same code, different outputs.} LLM-based operators are stochastic. Running the same prompt twice may yield different results.
  \item \textbf{Behavior depends on the environment.} Model version, decoding parameters, retrieval corpus, and tool availability all affect output, yet none are captured in a file tree.
  \item \textbf{``Did it get better?'' is a statistical question.} Comparing two program versions requires running evaluation suites and comparing metric distributions---not reading a text diff.
  \item \textbf{Agents produce patches, not just humans.} Provenance matters: which agent, which model, which prompt generated a change, and whether the result actually passes tests.
  \item \textbf{Merge can silently regress behavior.} Two branches may merge cleanly at the text level while producing outputs worse than either parent.
\end{itemize}

\noindent None of this is hypothetical. It is the default behavior of an increasing share of software development, and it motivates a different kind of version control---one that tracks not just \emph{what the files are} but \emph{what the transformation does} and \emph{whether it got better}.

We introduce \textsc{Morph}, a system that extends version control to \emph{certified behavior}. \textsc{Morph} preserves Git's core architectural principles---content-addressed immutable objects organized in a Merkle DAG---while adding first-class representations for transformation programs, evaluation contracts, execution evidence, and behavioral merge gating. A \textsc{Morph} commit goes beyond a file snapshot: it is a claim that a specific program, tested against a specific evaluation suite, achieved specific metric scores, backed by immutable execution receipts.

The core bet is simple: when software involves stochastic, effectful transformations, merge cannot be a purely syntactic operation. It has to be a \emph{behavioral} one too. \textsc{Morph} makes this concrete by requiring that any merged program dominate the behavioral claims of both its parents---or the merge fails.

The contributions of this paper are:

\begin{enumerate}[nosep]
  \item A formal model of transformation programs as effectful computations supporting sequential composition, parallel composition, and multi-agent attribution (\Cref{sec:programs}).
  \item An evaluation-contract formalism defining behavioral equivalence, certificate vectors, and a dominance preorder over programs (\Cref{sec:eval}).
  \item A merge semantics based on dominance of the join of parent certificates, with a proof that this preserves monotonic behavioral improvement (\Cref{sec:merge}).
  \item A concrete system design (\textsc{Morph} v0) implementing these ideas as a content-addressed object store with CLI tooling (\Cref{sec:system}).
\end{enumerate}

\section{Background and Related Work}
\label{sec:related}

\paragraph{Version control systems.}
Git~\citep{torvalds2005git} and its predecessors track deterministic source code as content-addressed snapshots organized in a Merkle DAG~\citep{merkle1988digital}. Commits are snapshots of file trees; branches diverge and reconcile via three-way text merge. This architecture has worked well for decades, but it provides no mechanism for reasoning about program \emph{behavior}---only about source \emph{text}. Darcs~\citep{roundy2005darcs} models patches algebraically, and Angiuli et al.~\citep{angiuli2014homotopical} give patches a homotopy-theoretic treatment that enables formal reasoning about commutativity and conflict. Both share our interest in principled merge semantics but operate at the syntactic level. Swierstra and L\"{o}h~\citep{swierstra2014semantics} demonstrated that separation logic can provide formal semantics for version control, showing that VCS operations can be modeled using program logics for mutable state. Their work establishes the precedent that version control deserves formal foundations, but their semantics reason about file content, not program behavior.

\paragraph{ML experiment and data versioning.}
DVC~\citep{dvc2020} extends Git to track large datasets and ML pipelines, treating data as versioned artifacts alongside code. MLflow~\citep{zaharia2018mlflow} provides experiment tracking, model packaging, and metric logging across training runs. Both address reproducibility in machine learning but focus on model training workflows rather than the composition and behavioral certification of transformation programs.

\paragraph{LLM evaluation frameworks.}
The evaluation landscape has grown rapidly alongside foundation model capabilities. HELM~\citep{liang2023helm} introduced holistic multi-metric evaluation, establishing the principle that single-score benchmarks are insufficient. SWE-bench~\citep{jimenez2024swebench} demonstrated execution-based evaluation of LLM-generated code changes against real-world test suites. Chatbot Arena~\citep{chiang2024chatbotarena} introduced pairwise human preference comparison at scale. These frameworks all take behavioral evaluation seriously, but they treat evaluation as an external measurement process rather than integrating it into the version-control lifecycle.

\paragraph{Prompt versioning and observability platforms.}
A growing class of commercial tools---including PromptLayer, Braintrust, LangSmith, and Humanloop---provide prompt versioning, A/B testing, evaluation pipelines, and deployment gating. Braintrust~\citep{braintrust2024} is notable for using content-addressable prompt IDs and CI-integrated evaluation that can block code merges when quality scores drop. These platforms reflect a real practitioner need: gating changes on behavioral evidence. But they differ from \textsc{Morph} in structural ways: they are centralized SaaS platforms, they version individual prompts rather than transformation program graphs, they lack formal merge semantics (merge blocking is a CI side-effect, not a first-class VCS operation), and they do not produce a decentralized, content-addressed evidence graph. They version prompts; \textsc{Morph} versions \emph{certified behavioral claims about transformation programs}.

\paragraph{LLM application frameworks.}
Frameworks such as LangChain~\citep{chase2022langchain} provide abstractions for composing LLM calls, retrieval, and tool use into pipelines, while observability platforms like LangSmith and Langfuse offer tracing and prompt versioning. These tools treat prompts as deployable artifacts with version histories but lack formal behavioral contracts, merge gating as a VCS primitive, or the immutable evidence graphs that \textsc{Morph} provides.

\paragraph{Agent frameworks.}
The ReAct paradigm~\citep{yao2023react} demonstrated that interleaving reasoning and action in LLMs produces more effective agent behavior. Multi-agent systems~\citep{wang2024survey} further expand the space of transformation programs to include concurrent, communicating agents. As agents become more capable and autonomous, the need for version control that tracks agent-authored changes with full provenance and behavioral certification becomes harder to ignore.

\paragraph{Effectful computation.}
Our formalization of programs as effectful computations $P_E : S \to F(S)$ draws on the tradition of monadic effects in programming language theory~\citep{moggi1991notions, wadler1995monads}. The short version: real-world computations do not just produce values---they also produce side effects (randomness, IO, logging, failures). A \emph{monad} gives you a structured way to compose such computations sequentially and in parallel without losing track of those effects. The monadic framework provides algebraic laws guaranteeing that pipeline composition is well-defined, and the zip operation handles parallel composition.

\paragraph{Reproducibility and provenance.}
Concerns about computational reproducibility~\citep{peng2011reproducible} motivate our treatment of environment recording and behavioral (rather than byte-level) reproducibility. In the LLM setting, strict output determinism is often impossible; \textsc{Morph} replaces it with the weaker but achievable guarantee that behavioral contracts can be re-certified.

\section{Programs as Effectful Transformations}
\label{sec:programs}

\subsection{State and Environment}

\textsc{Morph} models workspace state as a structured product:
\begin{equation}
  S = D \times C \times M,
\end{equation}
where $D$ is a document tree (code, datasets, artifacts), $C$ is execution context (intermediate results, caches), and $M$ is metadata (provenance references, run identifiers)---essentially a typed workspace struct.

Programs depend on an environment
\begin{equation}
  E = (\text{runner}, \text{model\_id}, \text{decoding\_params}, \text{toolchain}, \ldots),
\end{equation}
which is recorded because reproducibility is defined relative to it. The same program run under a different model version or with different decoding parameters may produce different results---and that difference matters for certification.

\subsection{Program Type}

For a fixed environment $E$, a program has type
\begin{equation}
  P_E : S \to F(S),
\end{equation}
where $F$ is an effect functor---a wrapper that carries side effects (randomness, IO, traces, failures, retries) alongside the result. A program takes a workspace state and returns a new workspace state \emph{plus} a record of everything that happened along the way.

A program is represented concretely as a directed acyclic graph (DAG) of typed operators:

\begin{definition}[Program]
A \emph{program} $P = (V, E_{\text{edges}}, \kappa, \rho)$ consists of a set of nodes $V$, directed edges $E_{\text{edges}} \subseteq V \times V$ forming a DAG, a kind function $\kappa : V \to \{\texttt{prompt\_call}, \texttt{tool\_call}, \texttt{retrieval}, \texttt{transform}, \texttt{identity}\}$, and a reference function $\rho : V \to \mathcal{H} \cup \{\bot\}$ mapping nodes to content-addressed blob hashes (e.g., prompt templates, tool schemas).
\end{definition}

Each edge is typed as either \texttt{data} (output flows as input) or \texttt{control} (execution ordering without data dependency). Human edits fit naturally: a human produces a patch blob, and the program includes a node \texttt{apply\_patch(patch\_blob)}. This makes manual coding a first-class, reproducible, reviewable operation within the same framework.

\subsection{Composing Programs}

To compose effectful computations---to chain one transformation into another---we need $F$ to be a \emph{monad}, providing two operations:
\begin{align}
  \text{pure} &: A \to F(A), \\
  \text{bind} &: F(A) \to (A \to F(B)) \to F(B).
\end{align}

\noindent \texttt{pure} wraps a plain value into the effect context; \texttt{bind} chains two computations by running the first, unwrapping the result, and feeding it to the second. If you have used \texttt{Promise.then()} or \texttt{flatMap}, you have used bind.

Sequential composition of programs follows directly:
\begin{equation}
  (Q \circ P)(a) = \text{bind}(P(a),\; Q).
\end{equation}

\noindent The standard monad laws ensure two things engineers care about:
\begin{itemize}[nosep]
  \item \textbf{Associativity:} $(R \circ Q) \circ P = R \circ (Q \circ P)$ --- you can restructure a pipeline without changing what it does.
  \item \textbf{Identity:} There exists an identity program $I$ such that $I \circ P = P \circ I = P$ --- there is a meaningful no-op.
\end{itemize}

\noindent This matters in practice: it is the mathematical reason that DAG scheduling and pipeline refactoring preserve meaning.

\subsection{Parallel Composition}

For parallel branches---concurrent prompt variants, agent swarms, independent experiments---we need a way to run two effectful computations independently and combine their results. This requires a zip operation:
\begin{equation}
  \text{zip}_{A,B} : F(A) \times F(B) \to F(A \times B),
\end{equation}
enabling the parallel composition of programs:
\begin{equation}
  (P \otimes Q)(a, c) = \text{zip}(P(a),\; Q(c)).
\end{equation}

\noindent This is \texttt{Promise.all()} or independent sampling, lifted to a type-level operation. For the common case where both branches consume the same input state, we compose with the diagonal $\Delta(s) = (s, s)$ and an explicit join step $J : S \times S \to F(S)$:
\begin{equation}
  \text{branch}(P, Q) = J \circ (P \otimes Q) \circ \Delta.
\end{equation}

\noindent This mirrors real systems: branches generate candidates; a join step selects or merges them.

\subsection{Multi-Agent Programs}

When multiple agents contribute to a single program---one agent generating retrieval logic, another writing prompt templates, a third composing the overall pipeline---the program DAG naturally records \emph{what} each agent did. We formalize this with an attribution function.

\begin{definition}[Attributed Program]
An \emph{attributed program} extends a program with an attribution function $\alpha : V \to \mathcal{A} \cup \{\bot\}$, where $\mathcal{A}$ is a set of agent identifiers and $\bot$ denotes unattributed nodes. For a single-agent or human-authored program, $\alpha$ is constant.
\end{definition}

Attribution composes naturally with both sequential and parallel composition. For $Q \circ P$, the attribution of the composite is the union of both attribution functions over disjoint node sets. For $P \otimes Q$, similarly. The join step $J$ in a branch composition may be attributed to a coordinating agent or left unattributed.

\paragraph{Certification is holistic.}
A critical property of the current framework: certification operates on the \emph{composed program}, not on individual contributions. The certificate vector $\text{cert}_T(P, E, s_0) \in V_T$ is a property of the program as a whole. There is no natural decomposition
\begin{equation}
  \text{cert}_T(P, E, s_0) = \bigoplus_{a \in \mathcal{A}} \text{cert}_T(P|_a, E, s_0)
\end{equation}
because the behavioral contribution of agent $a$'s nodes generally depends on the context provided by other agents' nodes. The evaluation suite tests the \emph{joint} result.

This is a deliberate design choice with a known consequence: \textsc{Morph} can certify that a multi-agent program meets its behavioral contract, and it can record which agent contributed which operators, but it cannot attribute \emph{credit} or \emph{blame} to individual agents from the certificate alone. Decomposing joint performance into per-agent contributions requires additional machinery---counterfactual evaluation (running the program with one agent's contributions ablated), Shapley-value-style credit assignment, or evaluation suites specifically designed to test modular contributions.

\paragraph{Distributed vs.\ cooperative multi-agent work.}
The framework distinguishes two multi-agent patterns:
\begin{itemize}[nosep]
  \item \textbf{Distributed:} Agents work on separate branches, each independently certified. Merge-as-dominance (\Cref{sec:merge}) ensures neither branch's behavioral gains are silently lost. This case is structurally identical to two humans on two branches and requires no new theory.
  \item \textbf{Cooperative:} Multiple agents contribute to a single program. The parallel composition operator $P \otimes Q$ with a join step models orchestrated parallelism---two agents generating candidates, a join step combining results. Attribution records provenance; certification evaluates the composed result.
\end{itemize}

\noindent Real-time concurrent edits to shared state (CRDT-style coordination) live below the VCS layer. Agents coordinate however they coordinate, produce a result, and that result is committed and certified by \textsc{Morph}.

\section{Evaluation Contracts and Behavioral Order}
\label{sec:eval}

\subsection{Evaluation Suites}

An evaluation suite $T$ defines an evaluator that is itself an effectful computation:
\begin{equation}
  \text{Eval}_{T,E} : S \to F(\text{Obs}_T),
\end{equation}
where $\text{Obs}_T$ includes raw metric samples, test execution logs, timing data, and any information needed for certification. Note that evaluation is itself effectful---running tests, compiling code, judging outputs with another model, collecting human ratings all involve side effects and potential failures. Each metric $m \in T$ specifies a scoring function, an ordering direction (maximize or minimize), and a certification method (e.g., lower confidence bound, quantile guarantee, deterministic pass/fail).

\textsc{Morph} supports two complementary forms of evaluation:

\begin{itemize}[nosep]
  \item \textbf{Artifact evaluation:} Evaluate the produced outputs---code correctness via unit tests, integration tests, static analysis, benchmarks.
  \item \textbf{Process evaluation:} Evaluate properties of the transformation itself---cost, latency, tool-usage constraints, policy compliance.
\end{itemize}

\noindent Both are meaningful for merges. You might require artifact correctness (tests pass) \emph{and} process bounds (cost under \$X, no forbidden tool calls).

To prevent programs from trivially satisfying their own tests, each evaluation suite declares a \emph{fixture source} for its test data: \texttt{candidate-sourced} (from the produced tree), \texttt{base-sourced} (from a specified baseline), \texttt{pinned} (immutably by hash), or \texttt{external}. Without this, a program could modify the tests that evaluate it and claim perfect scores.

\subsection{Certificate Vectors}

For an evaluation suite $T$ with metrics $\{m_1, \ldots, m_k\}$, define the certificate space:
\begin{equation}
  V_T = \prod_{m \in T} V_m,
\end{equation}
with componentwise ordering:
\begin{equation}
  x \leq_T y \iff \forall m \in T,\; x_m \leq_m y_m.
\end{equation}

\noindent A certificate vector is just a list of scores, one per metric. One vector dominates another if it is at least as good on \emph{every} metric (respecting direction---higher for accuracy, lower for latency).

A certification procedure yields a certificate vector:
\begin{equation}
  \text{cert}_T(P, E, s_0) \in V_T.
\end{equation}

\begin{definition}[Behavioral Dominance]
Program $Q$ \emph{dominates} program $P$ under suite $T$, environment $E$, and initial state $s_0$, written $P \preceq_{T,E,s_0} Q$, if and only if:
\begin{equation}
  \text{cert}_T(P, E, s_0) \leq_T \text{cert}_T(Q, E, s_0).
\end{equation}
\end{definition}

\begin{proposition}
Behavioral dominance $\preceq_{T,E,s_0}$ is a preorder (reflexive and transitive).
\end{proposition}
\begin{proof}
Reflexivity follows from reflexivity of $\leq_m$ for each metric. Transitivity follows from transitivity of $\leq_m$ applied componentwise.
\end{proof}

\noindent This turns ``did it get better?'' from a subjective judgment into a comparison with a well-defined answer.

\section{Commits and Merge}
\label{sec:merge}

\subsection{Commits as Behavioral Certificates}

A Git commit says: ``here is what the files looked like.'' A \textsc{Morph} commit says: ``here is a program, the evaluation suite it was tested against, and the scores it achieved---with receipts.''

\begin{definition}[Commit]
A \textsc{Morph} commit is a tuple
\begin{equation}
  c = (\text{tree\_hash},\; \text{program\_id},\; T,\; v,\; \text{parents},\; \text{env\_constraints},\; \text{evidence\_refs}),
\end{equation}
where $\text{tree\_hash}$ is the root hash of the working directory tree (same role as Git's tree object), $\text{program\_id}$ is the content hash of a program DAG, $T$ is an evaluation suite (by hash), $v \in V_T$ is the certified metric vector, $\text{parents}$ are parent commit hashes forming the Merkle DAG, $\text{env\_constraints}$ records the environment under which certification was obtained, and $\text{evidence\_refs}$ are hashes of supporting run and trace objects.
\end{definition}

The certified metric vector $v$ is not decorative---it is the merge contract. When the program and evaluation suite are unspecified, they default to the identity program and empty suite, making \textsc{Morph} usable as a plain VCS. History is immutable and verifiable, preserving Git's core property, but upgrading the meaning of what a commit represents.

\subsection{Merge as Join of Behavioral Requirements}

Git merge reconciles text diffs. \textsc{Morph} merge reconciles behavioral requirements:

\begin{enumerate}[nosep]
  \item \textbf{Structural merge} of program graphs (analogous to three-way text merge).
  \item \textbf{Union of evaluation contracts.} If parents have suites $T_1$ and $T_2$, the merge suite is $T = T_1 \uplus T_2$ (disjoint union by metric identifier). This prevents metric definitions from silently colliding.
  \item \textbf{Join of certificates.} Embed each parent certificate into $V_T$ and compute:
  \begin{equation}
    v_{\text{req}} = \text{embed}(v_1) \sqcup \text{embed}(v_2),
  \end{equation}
  where $\sqcup$ is the componentwise least upper bound (the best of both parents on every metric).
  \item \textbf{Dominance check.} The merged program $R$ is valid if:
  \begin{equation}
    \text{cert}_T(R, E, s_0) \geq_T v_{\text{req}}.
    \label{eq:merge-dom}
  \end{equation}
\end{enumerate}

\begin{theorem}[Merge Monotonicity]
\label{thm:monotone}
If a merge commit $c_m$ satisfies \Cref{eq:merge-dom}, then the merged program $R$ dominates both parent programs under the unified evaluation contract. Formally, for parent commits $c_1 = (\ldots, T_1, v_1, \ldots)$ and $c_2 = (\ldots, T_2, v_2, \ldots)$:
\begin{equation}
  \text{cert}_{T_1}(R, E, s_0) \geq_{T_1} v_1 \quad \text{and} \quad \text{cert}_{T_2}(R, E, s_0) \geq_{T_2} v_2.
\end{equation}
\end{theorem}

\begin{proof}
The merge certificate satisfies $\text{cert}_T(R, E, s_0) \geq_T v_{\text{req}}$. Since $v_{\text{req}} = \text{embed}(v_1) \sqcup \text{embed}(v_2)$, and the embedding preserves metric values while extending with bottom elements, projecting onto the metrics of $T_1$ yields $\text{cert}_{T_1}(R, E, s_0) \geq_{T_1} v_1$, and symmetrically for $T_2$.
\end{proof}

This means merge can fail with zero text conflicts. If parent A achieved accuracy 0.95 and parent B achieved latency \mbox{$p_{95}$ = 1.2s}, the merged program must meet \emph{both}. If it cannot, you get a \emph{behavioral conflict}---analogous to a text conflict in Git, but at the semantic level. A merge that silently drops accuracy or blows up latency is the actual failure mode, and traditional VCS lets it through without complaint.

\section{Object Model and System Design}
\label{sec:system}

\subsection{Content-Addressed Immutable Objects}

Like Git, \textsc{Morph} stores all objects as content-addressed, immutable nodes in a Merkle DAG, using SHA-256. The object types extend Git's \texttt{blob}/\texttt{tree}/\texttt{commit} with new first-class entities:

\begin{table}[t]
\centering
\caption{Object types in \textsc{Morph} compared to Git. Items marked with $\star$ are new to \textsc{Morph}; others generalize Git equivalents.}
\label{tab:objects}
\begin{tabular}{@{}llp{6.5cm}@{}}
\toprule
\textbf{Object} & \textbf{Git Analog} & \textbf{Description} \\
\midrule
Blob & blob & Atomic content: prompt templates, tool schemas, configs. Open \texttt{kind} field. \\
Tree & tree & Structured grouping of objects (directory analog). \\
Program$^\star$ & --- & DAG of typed operators with data/control edges. \\
EvalSuite$^\star$ & --- & Versioned evaluation definition: cases, metrics, thresholds, fixture sources. \\
Commit & commit & Tree hash $+$ program hash $+$ eval contract $+$ certified metric vector $+$ evidence refs. \\
Run$^\star$ & --- & Execution receipt: environment, inputs, outputs, metrics, trace ref, agent identity. \\
Trace$^\star$ & --- & Sequence of typed, addressable events within a run. \\
Annotation$^\star$ & --- & Metadata on any object without changing its hash. \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Runs and Traces as First-Class Evidence}

Execution evidence is immutable and first-class. A \textbf{Run} records exactly what happened: the program hash, full environment (model identifier, version, decoding parameters, toolchain), input state, output artifacts, observed metrics, a trace reference, and the executing agent's identity. A \textbf{Trace} is the detailed execution record---a sequence of typed events, each with a unique identifier enabling fine-grained annotation.

Runs never modify commits. They are evidence. This separation means that multiple runs can accumulate evidence for or against a program without rewriting history.

\subsection{Annotations}

\textbf{Annotations} attach metadata to any object (or a specific event within a trace) without altering its hash. They enable human ratings, bookmarks, tags, cross-references, and provenance overlays. Annotations are the extensibility primitive: higher-level tools can layer rich metadata onto the immutable object graph without requiring changes to core object types.

\subsection{Separation of Concerns: \textsc{Morph} Does Not Execute}

An important design constraint: \textsc{Morph} does not execute programs or run evaluations. All LLM calls, tool execution, and test runs happen in external tools---IDEs, agents, CI systems. \textsc{Morph}'s role is to:
\begin{itemize}[nosep]
  \item \textbf{Store} immutable content-addressed objects.
  \item \textbf{Record} execution evidence that external tools report (runs, traces, metrics).
  \item \textbf{Version} programs with behavioral contracts (commits).
  \item \textbf{Gate} merges on metric dominance.
\end{itemize}

Commands like \texttt{morph run record} and \texttt{morph eval record} \emph{ingest} results; they do not run anything. This keeps the VCS layer thin, composable, and out of the way.

\subsection{CLI Design}

The CLI mirrors Git where possible:

\smallskip
\begin{center}
\begin{tabular}{@{}ll@{}}
\texttt{morph init} & Initialize repository \\
\texttt{morph add .} & Stage working-space changes \\
\texttt{morph commit -m "msg"} & Commit with eval contract \\
\texttt{morph merge <branch>} & Behavioral merge (eval-gated) \\
\texttt{morph run record <file>} & Ingest execution receipt \\
\texttt{morph eval record <file>} & Ingest evaluation results \\
\texttt{morph annotate <hash>} & Attach metadata \\
\texttt{morph log} / \texttt{status} & Inspect history and state \\
\end{tabular}
\end{center}

\section{Axioms}
\label{sec:axioms}

\textsc{Morph} requires the following minimal axioms for coherence:

\begin{axiom}[Immutable Content-Addressed Objects]
All primary objects are immutable and content-addressed by cryptographic hash.
\end{axiom}

\begin{axiom}[Evidence Does Not Rewrite History]
Runs and evaluation evidence never mutate prior commits; new claims are new commits.
\end{axiom}

\begin{axiom}[Effect Monad for Sequencing]
There exists an endofunctor $F$ that is a monad, enabling associative sequential composition and identity programs.
\end{axiom}

\begin{axiom}[Product State Spaces]
State spaces support products $A \times B$ and a unit object $1$.
\end{axiom}

\begin{axiom}[Zip for Parallelism]
$F$ provides a lawful zip: $\text{zip}_{A,B} : F(A) \times F(B) \to F(A \times B)$.
\end{axiom}

\begin{axiom}[Evaluation Suites are Explicit Contracts]
An EvalSuite defines metrics (with ordering direction), fixture sources, and a certification rule.
\end{axiom}

\begin{axiom}[Certificates are Comparable]
Certified metric vectors live in a product preorder with componentwise dominance.
\end{axiom}

\begin{axiom}[Merge is Dominance of Joined Requirements]
A merge commit must certify dominance over the join of parent certificates under the union of their contracts.
\end{axiom}

\begin{axiom}[Explicit Environment Recording]
Every run records the environment $E$ needed to interpret evidence.
\end{axiom}

\begin{axiom}[Behavioral Reproducibility]
Reproducibility means the behavioral claim can be re-certified under declared environment constraints, not byte-identical outputs.
\end{axiom}

\section{Discussion}
\label{sec:discussion}

\paragraph{Relationship to continuous integration.}
CI pipelines already run tests on proposed changes. \textsc{Morph} formalizes what CI enforces informally: a behavioral contract that merges must satisfy. Tools like Braintrust already do this informally through CI integration; \textsc{Morph} makes the contract, the evidence, and the dominance check first-class versioned objects rather than ephemeral pipeline logs. Failed CI runs disappear; \textsc{Morph} runs persist as immutable evidence.

\paragraph{Agent accountability.}
When an AI agent generates a code patch, \textsc{Morph} records the agent's identity, the model version, the prompt that produced the change, and the full execution trace. If the patch is committed, the commit's evaluation contract certifies its behavioral properties. This creates an accountability chain from agent action to certified outcome---something that does not exist in current agent-driven development workflows.

\paragraph{Statistical sophistication.}
The v0 system implements minimal statistical machinery: mean aggregation, worst-case scores, percentile thresholds, and lower confidence bounds. Richer certification methods---Bayesian comparison, two-sample equivalence testing, human-in-the-loop evaluation pipelines---are anticipated by the framework's abstract evaluation interface but deferred to future work.

\paragraph{Scalability and distribution.}
Because all objects are content-addressed and immutable, the architecture extends naturally to distributed settings (remote push/pull, federated verification). The v0 implementation is local-only; a remote protocol is planned future work.

\paragraph{Complementarity with Git.}
\textsc{Morph} does not replace Git. Deterministic source code remains well-served by Git. \textsc{Morph} handles the transformation programs, evaluations, and behavioral contracts that Git cannot express. The two systems coexist in the same repository: Git for source snapshots, \textsc{Morph} for behavioral versioning. When program and evaluation suite fields are left empty, a \textsc{Morph} commit degenerates into a standard tree-hash commit, so you do not have to use the behavioral features until you need them.

\paragraph{Multi-agent attribution.}
The current formalization treats programs as units for certification: the certificate vector is a property of the composed program, not of individual operator contributions (\Cref{sec:programs}). When multiple agents contribute to a single program, the attribution function records which agent authored which operators, but the dominance preorder operates on the whole. Decomposing a certificate into per-agent credit is analogous to the credit assignment problem in cooperative multi-agent reinforcement learning~\citep{wang2024survey}---the joint reward is observable, but individual contributions require counterfactual reasoning. For distributed multi-agent work (separate branches), this is not a limitation: each branch carries its own certification. For tightly coupled cooperative work---where agent A's contribution only makes sense in the context of agent B's---the evaluation suite must test the composed result. Future work could introduce modular evaluation suites that test sub-programs in isolation, enabling finer-grained attribution, or integrate Shapley-value-style decomposition into the certification framework.

\paragraph{Limitations.}
The current formalization assumes evaluation suites are honest and that certified metrics actually measure something meaningful. The dominance preorder is only as good as the contracts underneath it---bad evals produce meaningless certificates. Additionally, the structural merge of program graphs (step 1 of merge) is under-specified relative to three-way text merge, which has decades of tooling behind it; developing robust program-graph merge algorithms is important future work.

% ============================================================
\section{Conclusion}
\label{sec:conclusion}

We have presented \textsc{Morph}, a version control system for transformation programs that extends Git's content-addressed Merkle DAG with first-class support for behavioral certification. Programs are formalized as effectful computations, evaluation suites as behavioral contracts, and merge as dominance over joined certificate requirements.

The bet is that when software involves stochastic, effectful transformations, ``what changed'' becomes a behavioral question, not a syntactic one---and version control has to evolve to match. Git versioned deterministic source code. \textsc{Morph} versions certified behavior, with receipts.
