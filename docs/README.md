# Morph
## Version Control for Prompt Programs

---

# Why Morph Exists

Git changed software development by making source code:

- Reproducible
- Traceable
- Distributed
- Collaborative
- Mergeable

But Git assumes something fundamental:

> Code is deterministic text.

Modern AI systems break that assumption.

Today, software is increasingly built from:

- Prompt programs
- LLM calls
- Retrieval pipelines
- Tool-using agents
- Model-dependent behavior
- Stochastic outputs
- Autonomous code generation

In this world:

- The same prompt may produce different outputs.
- The same code may be generated by different agent runs.
- Behavior depends on model versions.
- Evaluation is statistical, not binary.
- Agents may write more code than humans.

Git alone is no longer enough.

Morph exists to generalize version control to probabilistic, effectful systems.

---

# The Core Idea

Git versions files.

Morph versions **transformations**.

More precisely:

> Morph versions effectful prompt programs that transform structured document state.

A prompt program is a function P : S → F(S) — it takes structured state, runs through models and tools, and produces probabilistic output. F captures the computational effects (randomness, external calls, nondeterminism).

A Morph commit does not just freeze text.

It freezes a **behavioral identity** — a program together with its certified evaluation metrics.

---

# The Problem Git Cannot Solve

Consider an AI-driven system:

- A prompt generates a refactor.
- An agent edits code.
- Retrieval pulls documents.
- A model upgrade changes output distribution.
- Evaluation scores fluctuate statistically.

Git can tell you:

- Which files changed.

It cannot tell you:

- Which prompt produced this patch.
- Which model version was used.
- Whether behavior improved or regressed.
- Whether outputs are statistically stable.
- Which agent created this change.
- Whether the merge preserved evaluation guarantees.

Morph can.

# What Morph Provides

## 1. Behavioral Version Control

Morph defines equality behaviorally.

Two prompt programs are equivalent if they produce statistically indistinguishable outputs under a declared evaluation suite.

Commits represent stabilized behavioral identities — not just text snapshots. Each commit records the program, the evaluation suite, and the observed metric scores.

---

## 2. Reproducibility (Reimagined)

Morph supports multiple levels of reproducibility:

- Deterministic replay (when possible)
- Distributional stability
- Property preservation
- Evaluation contract satisfaction

Reproducibility is defined by behavior, not bytes. A program is reproducible if re-running it satisfies its declared evaluation contract.

---

## 3. Execution Receipts

Every run in Morph produces a first-class **Run object** that records:

- Prompt program
- Environment (model, parameters, toolchain)
- Inputs
- Outputs
- Metrics
- Trace (typed, addressable events)
- Agent identity (if applicable)

Commits are claims.

Runs are receipts.

---

## 4. Decentralized Collaboration

Morph is a distributed version control system:

- Content-addressed objects
- Merkle DAG history
- Forking
- Branching
- Merging
- Multiple remotes

No central authority required.

---

## 5. Git-Compatible Workflows

Morph preserves familiar commands and mental models:

- `commit`
- `branch`
- `merge`
- `rebase`
- `cherry-pick`
- `tag`
- `rollup` (squash)
- hooks
- CI/CD integration

If you know Git, you can use Morph.

The difference is semantic, not operational.

---

## 6. Agent-Native by Design

Morph assumes agents will:

- Write code
- Modify prompt programs
- Run evaluation loops
- Generate large trace trees

Morph provides:

- Agent identity tracking in every Run
- Policy reference versioning
- Evaluation gating before merge
- Evidence-backed commits
- Rollups that distill exploratory traces into stable programs

Morph is the coordination and accountability layer for agentic software development.

---

## 7. Prompts as First-Class Objects

Prompts in Morph are not just text in files.

They are canonical objects in the content-addressed store.

However, they can be materialized as files for:

- Diffing
- Code review
- Pull requests

The object store is canonical.
Filesystem views are projections.

---

## 8. Working Space and Commit Space

Morph separates exploration from stabilization:

- **Working space**: filesystem-level prompt evolution, agent experimentation, trace accumulation
- **Commit space**: evaluation-certified behavioral identities in the content-addressed store

Rollup collapses exploratory history into stable identities. Traces are never deleted — they may be summarized.

---

## 9. Behavioral Merge

Merge in Morph is not purely structural.

A merge commit must **dominate** both parents — its observed evaluation scores must meet or exceed what both parents achieved.

If dominance cannot be demonstrated, merge fails. This prevents behavioral regression from entering the commit history.

---

## 10. Annotations

Morph objects are immutable, but the metadata around them is not static.

**Annotations** attach feedback, bookmarks, tags, notes, and cross-references to any object — without altering its hash. Annotations are themselves content-addressed and immutable: adding metadata never rewrites history.

This is the extensibility primitive of the Morph object model. Higher-level tools can layer rich, domain-specific metadata onto Programs, Runs, Traces, and individual trace events without requiring changes to Morph's core. Human feedback (ratings, curation signals) enters the system through Annotations and can be aggregated into evaluation metrics, closing the loop between human judgment and the behavioral preorder.

# The Vision

Git versioned deterministic source code.

Morph versions stochastic meaning.

Git tracks files.

Morph tracks transformations.

As software becomes:

- Model-mediated
- Probabilistic
- Agent-authored
- Behaviorally evaluated

Version control must evolve.

Morph is that evolution.
