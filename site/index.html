Research Project

Version control when pipelines are probabilistic

Git tracks files where identity is byte equality and merge is text reconciliation. None of that holds when you're building with LLMs. Morph records what your pipelines did and whether they got better.

View on GitHub Read the Paper

The Problem

Git wasn't built for this

LLM applications aren't source code. They're stochastic, environment-dependent transformation pipelines. Git's assumptions break in five specific ways.

~

Same code, different outputs

Run the same prompt twice and you get different answers. That's not a bug — it's how LLMs work. Git's assumption that identity means byte equality doesn't hold.

···

Behavior depends on the environment

Model version, sampling settings, retrieval corpus, tool availability — all shape the output. None of it is in the file tree.

↑↓

"Did it get better?" is a statistical question

You can't read a diff and know whether the pipeline improved. You have to run it, score it, and compare.

>_

Agents produce patches, not just humans

Which agent, which model, which prompt generated a change — and whether it passed any checks — matters. Git has no place for any of that.

×

Merge can silently regress behavior

Two branches can merge cleanly at the text level while the resulting pipeline performs worse than either parent.

The Solution

Morph extends Git with behavioral evidence

Same content-addressed Merkle DAG. Same hash-of-contents identity. Three additions that make version control work for LLM pipelines.

Pipelines as first-class objects

A pipeline is a DAG of typed steps — prompt calls, tool invocations, retrieval lookups, deterministic transforms, review decisions. Each node records what ran, who ran it, and in what environment. Morph versions the whole graph, not just the files.

A Morph commit vs. a Git commit // Git commit
tree      a3f2c...
parent    b71e0...
message   "fix retrieval"

// Morph commit
tree      a3f2c...
parent    b71e0...
message   "fix retrieval"
pipeline  d4e8a...
eval      f92b1...
scores    {accuracy: 0.94, latency_p95: 320ms}
evidence  [run:c81f, trace:e02a]

Evaluation suites as versioned contracts

"Better" is not implicit. Each evaluation suite declares its metrics, directions, fixture sources, and aggregation methods — all versioned and hashed. Scores are partially ordered: one pipeline dominates another only if it wins on every metric.

Improvement ordering // Pipeline Q improves on P when:

cert(Q) ≥ cert(P)  on every metric

// Example:
P: {accuracy: 0.91, latency: 340ms, cost: $0.03}
Q: {accuracy: 0.94, latency: 320ms, cost: $0.02}
↑ Q dominates P

R: {accuracy: 0.96, latency: 400ms, cost: $0.02}
↑ R vs P: incomparable (better accuracy, worse latency)

Runs and traces as permanent receipts

Every pipeline execution produces a Run — a permanent record of the pipeline hash, full environment, input state, output artifacts, scores, and actor identity. Inside each Run is a Trace: the sequence of every prompt call, tool invocation, and retry. Runs never modify commits. They're evidence that accumulates over time.

Merge with evidence // At merge time, Morph records:

parent_1_scores  {accuracy: 0.94, cost: $0.03}
parent_2_scores  {accuracy: 0.91, cost: $0.02}
reference_bar    {accuracy: 0.94, cost: $0.02}
merged_scores    {accuracy: 0.95, cost: $0.02}

// The merge improved. The record proves it.

How It Works

Git-style CLI, richer objects

Morph mirrors Git where possible. If you know Git, you know Morph. The CLI adds commands for recording runs and evaluation results.

# Initialize a Morph repository
$ morph init

# Stage and commit with eval scores
$ morph add .
$ morph commit -m "improve retrieval prompt"

# Record an execution receipt
$ morph run record run-output.json

# Record evaluation results
$ morph eval record metrics.json

# Merge with score recording
$ morph merge feature-branch

# Inspect history
$ morph log

Design Principles

What Morph assumes

Morph is built on a small set of axioms. These are the design constraints the system depends on — violate any of them and something breaks.

01

Content-addressed, immutable objects. Every object is identified by a hash of its contents. History cannot be tampered with.

02

Evidence does not rewrite history. Runs and evaluation results never mutate prior commits. New evidence produces new objects.

03

Pipeline steps compose cleanly. Sequential chaining and parallel execution are well-defined — like Promise.then() and Promise.all().

04

Evaluation suites are explicit contracts. "Better" is never implicit. Metrics, directions, fixture sources, and aggregation methods are all versioned and hashed.

05

Scores are partially ordered. One scorecard dominates another only if it wins on every metric. If A is more accurate but slower, they're incomparable.

06

Merge records scores from both parents. Every merge commit records what both parents achieved and what the merged pipeline achieved.

07

Environment is part of the record. Model version, sampling settings, toolchain — without this, scores from different environments aren't comparable.

08

Reproducibility means re-running the checks. You can't get identical outputs from an LLM. Reproducibility means re-running the evaluation and getting consistent aggregate scores.
