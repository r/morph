Changelog

What's changed in Morph

The three most recent releases. Morph is alpha — the on-disk format and CLI surface still move. Full history lives in CHANGELOG.md in the repo, and the unabridged record is the git log itself.

0.42.2 Latest 2026-05-02

morph init --git-init now lands on main on every host. The ref-mode contract treats main as the canonical default branch (it's what morph checkout, morph branch, morph push, and the merge breadcrumbs all assume), but the underlying git init shell-out picked up whatever the host's init.defaultBranch happened to be — master on a fresh GitHub-Actions runner that hadn't opted in to the git 2.28+ rename. Spec tests like spec_checkout_branch and the entire merge_* family then failed with pathspec 'main' did not match any file(s) known to git because git landed the very first commit on master.

Fixed

  • morph init --git-init passes -b main (anchored to morph_core::DEFAULT_BRANCH) so the initial branch matches morph's contract regardless of host config. The on-failure error message now also names the explicit invocation (git init -b main <path>) so a reader of the failure output can reproduce it without guessing.

Tests

  • Workspace (1187 tests) verified under a simulated CI environment (GIT_CONFIG_COUNT=1 GIT_CONFIG_KEY_0=init.defaultBranch GIT_CONFIG_VALUE_0=master) — 1187 / 1187 passing, 0 ignored. No regressions on hosts that already default to main.
0.42.1 2026-05-01

Mixed-authorship plumbing on the ref-mode commit rebuild. The last three previously-skipped acceptance specs unblock: ref-mode morph commit --from-run now folds the human author into commit.contributors with role = "human-author" alongside the agent's role = "primary", and the rebuild runs compute_human_edits against the mirror commit's flattened tree so the trace-claim-vs-staged-blob diff lands in human_edits just like standalone mode. Workspace test count: 1187 / 1187 passing, 0 ignored (down from 13 ignored at the start of the v0.41 line).

Fixed

  • Human author lands in commit.contributors with role = "human-author" on ref-mode morph commit --from-run. Earlier ref-mode commits showed only the agent in the structured contributors list.
  • human_edits records (trace claim) vs (staged blob) divergence: post-agent-edit when the human edits a path the agent already wrote; no-trace-record when the human authors a path the trace never references. Earlier ref-mode commits silently dropped this diff.

Changed

  • morph_core::fold_human_author_into_contributors is now public so the CLI can call it directly. Same recipe the standalone-mode commit path already used.

Tests

  • mixed_authorship.yaml: mixed_authorship_human_author_in_contributors_when_run_attached, mixed_authorship_variant_a_post_agent_edit_recorded, mixed_authorship_variant_b_no_trace_record_recorded — all three un-skipped and passing. version.yaml updated to expect 0.42.1.
0.42.0 2026-05-01

Reference-mode merge gets its evidence back. The merge rebuild path now propagates the user's eval suite, pipeline, and metric-retirement decisions onto the mirrored merge commit instead of leaving them as the empty placeholders sync_one_commit originally planted. Ten previously skipped acceptance specs unblock as a consequence — the merge gate has real evidence to enforce on every ref-mode merge, single-shot or stateful. Three skipped specs remain (mixed-authorship plumbing, scheduled for v0.42.1).

Added

  • morph_core::rebuild_merge_commit + MergeRebuildOpts. After sync_to_head_with_origin mirrors the new merge commit, the rebuild helper re-derives the commit using the MergePlan's auto-union eval suite, the user's --pipeline / --eval-suite / --metrics, and any retired metrics, then re-points the branch ref onto the rewritten hash.
  • MERGE_REF.json carries merge context across the conflict boundary. ReferenceMergeBreadcrumb now records the user's --pipeline, --eval-suite, --retire, and --retire-reason so morph merge --continue reproduces the same plan a single-shot merge would have used.
  • auto_detect_introduces_cases() in morph_core::annotate. morph commit (without --new-cases) now diffs the about-to-commit eval suite against the first-parent's suite and auto-records the case-id difference as an introduces_cases annotation.

Fixed

  • morph merge propagates the auto-union eval suite onto the merged commit; the dominance check now sees the right metrics on subsequent merges.
  • morph merge --retire <metric> writes a real review node attributing the retirement to the merge author and the optional --retire-reason "...".
  • morph merge --retire survives a textual conflict — the retire intent flows through MERGE_REF.json to morph merge --continue.
  • Weak merge metrics are rejected with the parent name on stderr (current: acc 0.9 → 0.87) instead of silently passing.

Tests

  • Ten previously-#[ignore]'d specs flip green: 3 in merge.yaml, 3 in merge_retire_review.yaml, 3 in merge_retire_stateful.yaml (the latter two also rewritten for v0.40+ stderr-shaped conflict UX), and commit_auto_detects_new_cases_added_to_default_suite. Workspace test count: 1184 / 1184 passing, 3 ignored. version.yaml updated to expect 0.42.0.
