Changelog

What's changed in Morph

The three most recent releases. Morph is alpha — the on-disk format and CLI surface still move. Full history lives in CHANGELOG.md in the repo, and the unabridged record is the git log itself.

0.44.0 Latest 2026-05-02

Commit ergonomics. Second release of the “make morph intuitive” effort. morph commit becomes a one-liner even when you want metrics: tell Morph your test suite once with morph config commit.test_command "cargo test --workspace" (or pytest, vitest, jest, go) and from then on plain morph commit shells out to the command, parses the output, writes a fresh LAST_RUN.json breadcrumb, and attaches the metrics + run as evidence — collapsing the eval-driven flow from two commands to one for the common case. --no-test opts out for a single commit, --rerun forces a fresh run when the breadcrumb is stale, and a failing test aborts the commit so red suites don't slip into the behavioral history.

Added

  • commit.test_command config key in .morph/config.json. Set with morph config commit.test_command "…"; read with morph config commit.test_command or --get; exit 1 when unset (matches the user.* convention). The string is POSIX-shell-split with shlex so quoted arguments survive ("pytest -k 'fast and not slow'"). Spec coverage: commit_runs_configured_test_command.yaml, config_commit_test_command.yaml.
  • morph commit --no-test — skip the configured auto-run for this commit without removing the configured command. Useful for chores or out-of-band certification.
  • morph commit --rerun — force a fresh test run even when the most recent breadcrumb is current. Use after an external state change makes cached metrics stale.

Changed

  • Failing tests abort the commit. A non-zero exit from commit.test_command is treated as evidence the code is not in a committable state. The failing run is still recorded (inspect with morph show <hash>), but the commit doesn't land. Override with --no-test.
  • README quickstart leads with the configured command. The recommended one-time setup is morph config commit.test_command "…"; the old two-step (morph eval run then morph commit) is preserved as the manual fallback for repos that prefer it.
  • docs/EVAL-DRIVEN.md §5 features the one-command flow as the primary path. The breadcrumb-pickup and explicit --from-run <hash> forms are demoted to subsections.
  • Tutorials updated. site/tutorials/getting-started.html §7 gains a "Skip the two-step?" callout pointing at the configured-command flow; site/tutorials/adding-to-git-project.html rewrites the post-staging commit example to use it.

Tests

  • Workspace 1203 / 1203 passing (1192 baseline + 6 new acceptance cases across the two new spec files + 5 new unit tests in morph-core/src/commit_config.rs). Cucumber 34 / 37 (3 skipped, 0 failed) — no change.
0.43.0 2026-05-02

Onboarding polish. First release of the multi-phase “make morph intuitive” effort. morph init now prints the active policy as the last line of init output (policy: relaxed (metrics optional) — tighten with morph policy init), so a fresh user knows the gate they're under without having to run morph policy show. morph --help gains a COMMANDS BY GROUP: section (ESSENTIAL / REMOTES / EVALS & METRICS / INSPECT / ADVANCED / INTEGRATIONS) so the day-one happy path is discoverable without scanning 46 alphabetical entries. The README now leads with one command (morph init + morph commit); the full surface is preserved in a reference section below. docs/EVAL-DRIVEN.md aligns with reality (reference-mode default has been permissive since v0.40) and leads with the LAST_RUN.json auto-pickup flow instead of the manual --from-run <hash> ritual.

Added

  • morph init prints the active policy in its last line, with a one-liner showing how to switch. Reference-mode shows relaxed; bare repos show strict with the loosening recipe. Spec coverage: init_prints_active_policy.yaml.
  • morph --help renders a grouped subcommand overview after the auto-generated list: ESSENTIAL / REMOTES / EVALS & METRICS / INSPECT / ADVANCED / INTEGRATIONS. Spec coverage: help_is_grouped.yaml.

Changed

  • README leads with one command. The quickstart is morph init followed by morph commit -m "..." and nothing else; the full command list moved to a reference section near the bottom. The "When you want more" section gradually introduces IDE recording, test-result attachment, and dominance gating as opt-in upgrades.
  • docs/EVAL-DRIVEN.md documents reality. §0 used to claim reference-mode morph init installed a strict default policy; in fact reference-mode has been writing a permissive one (with the git-hook carve-out) since v0.40. The doc now matches the code and points readers at morph policy init for opt-in strict mode.
  • §5 leads with auto-pickup. morph eval run followed by plain morph commit (the LAST_RUN.json breadcrumb attaches metrics automatically) is now the canonical flow; the manual --from-run <hash> ritual is preserved in a "When you need precision" section.

Tests

  • Workspace 1192 / 1192 passing (1187 baseline + 5 new acceptance cases across the two new spec files). Cucumber 34 / 37 (3 skipped, 0 failed) — no change. Pre-existing fmt drift across 55 workspace files cleared in a separate chore: cargo fmt --all baseline commit so this release lands on a fmt-clean base.
0.42.2 2026-05-02

morph init --git-init now lands on main on every host. The ref-mode contract treats main as the canonical default branch (it's what morph checkout, morph branch, morph push, and the merge breadcrumbs all assume), but the underlying git init shell-out picked up whatever the host's init.defaultBranch happened to be — master on a fresh GitHub-Actions runner that hadn't opted in to the git 2.28+ rename. Spec tests like spec_checkout_branch and the entire merge_* family then failed with pathspec 'main' did not match any file(s) known to git because git landed the very first commit on master.

Fixed

  • morph init --git-init passes -b main (anchored to morph_core::DEFAULT_BRANCH) so the initial branch matches morph's contract regardless of host config. The on-failure error message now also names the explicit invocation (git init -b main <path>) so a reader of the failure output can reproduce it without guessing.

Tests

  • Workspace (1187 tests) verified under a simulated CI environment (GIT_CONFIG_COUNT=1 GIT_CONFIG_KEY_0=init.defaultBranch GIT_CONFIG_VALUE_0=master) — 1187 / 1187 passing, 0 ignored. No regressions on hosts that already default to main.
